Principal Engineer Review

BigConfig — Area of Improvement Report

An architectural and code-quality assessment of the BigConfig Clojure/Babashka workflow & template engine, with prioritized, actionable refactors for the highest-impact issues.

Repository: big-config (main) Date: 2026-05-16 Scope: Entire repo — src, tests, templates, build, CI Method: Static review (no tooling executed) Surface: ~2,940 LOC src / 21 namespaces / 13 test files

Executive Summary

BigConfig is a well-conceived, data-oriented workflow engine. The core abstractions (->workflow, ->step-fn, pluggable multimethods, an opts map threaded through pure steps) are idiomatic and genuinely composable. The Redis event-sourcing store correctly implements optimistic concurrency, and the pure helpers (parse-args, prepare, merge-params) are clean and well-tested.

The problems are concentrated, not pervasive. Three substantive issues stand out: a hard-wired shell boundary that simultaneously creates injection fragility and leaves ~50% of namespaces (including the security-sensitive lock) untestable; hidden mutable state that undermines the otherwise-functional engine; and accumulated duplication / dead code from an unfinished deprecation. A fourth item — the directory-isolation key — is a documentation defect, not a bug: the deterministic paths are intentional and required for local-backend state continuity and create/delete pairing, but the docstring mis-sells that determinism as a concurrency guard. None require a rewrite — each has a contained, mechanical fix.

Detailed Findings

#1Low

Isolation-key docstring oversells deterministic paths as a concurrency guard

Location
workflow.clj:116–119 (namespace docstring); behavior at workflow.clj:477–496 (new-prefix)
Dimensions
Documentation · Correctness of intent (not of behavior)
Status
Behavior is intentional and correct; only the docstring needed narrowing — fixed in workflow.clj

Clarification (revised after maintainer input). new-prefix derives the workspace discriminator deterministically from :first-step. This is a deliberate design requirement, not a defect:

  • Local OpenTofu backend: terraform.tfstate lives in the rendered directory. A unique-per-run path would make every run start from empty state and never see previously-created resources — stable paths are mandatory here.
  • Dev iteration: a workflow always resolves to the same .dist/… path, so generated OpenTofu/Ansible can be inspected and re-run in place.
  • create/delete pairing: the two workflows intentionally share :first-step so delete resolves to the same state directory create used — which works because the hash is deterministic.

The only defect was the docstring at workflow.clj:116–119, which described the hash as "avoiding collisions when the same workflow is run concurrently." The hash is a workspace selector, not a concurrency primitive. Mutual exclusion is already a separate, explicit layer in the architecture — the lock step (big-config.lock, git-tag pessimistic lock). Concurrent runs of one workflow against one prefix are an explicit non-goal by design.

Before — docstring (workflow.clj:116–119)

> Note: ->workflow* calls new-prefix on the globals
> derived from opts, which stamps a UNIQUE hash into
> both ::prefix and ::object-prefix. Each invocation
> of the composite workflow therefore writes to an
> isolated directory and object path, AVOIDING
> COLLISIONS when the same workflow is run
> concurrently.

After — states intent + explicit non-goal

> Note: new-prefix derives a DETERMINISTIC
> discriminator from :first-step. Intentional: a
> workflow always resolves to the SAME dir/object so
> (a) generated code can be iterated on, (b) a local
> OpenTofu backend keeps terraform.tfstate across
> runs, (c) paired create/delete (shared :first-step)
> share state.
>
> Concurrency is an EXPLICIT NON-GOAL here; mutual
> exclusion is the job of the lock step
> (big-config.lock). For isolated parallel runs,
> override ::prefix / ::object-prefix explicitly.
Impact — Documentation / Correctness of intent: No behavior change. Removes the only thing that made an intentional, required design read as a race, and aligns the docstring with the actual concurrency layer (lock). Reclassified from Critical to Low after maintainer confirmation that deterministic paths are required for local-backend state continuity.
#2High

Shelling-out is hard-wired with string-built commands: no test seam & injection-fragile

Location
run.clj:20–29 (generic-cmd); callers in lock.clj, git.clj, unlock.clj, run.clj:31–40
Dimensions
Testability · Security & Robustness · Maintainability

The issue. Every external command flows through generic-cmd, which calls process/shell directly with a string assembled via format (e.g. (format "git tag -d %s" lock-name)). Two consequences: (a) there is no injection point for a fake runner, so anything that shells out can only be exercised by running real git/bash — which is precisely why 11 of 21 namespaces, including the security-sensitive big-config.lock, have no unit tests and the only coverage is a slow integration test that git init --bares and spawns clojure -M:test; (b) string interpolation into commands — some of which are literally bash -c '…' (run.clj:33) — is a latent shell-injection / tokenization-fragility class the moment any interpolated value contains whitespace or shell metacharacters.

Before

;; run.clj
(defn generic-cmd [& {:keys [opts cmd key shell-opts]}]
  (let [shell-opts (merge {:continue true :out :string
                           :err :string} shell-opts)
        proc (process/shell shell-opts cmd) ; direct
        opts (handle-cmd opts proc)]
    (if key
      (assoc opts key (-> (:out proc) str/trim-newline))
      opts)))

;; lock.clj — string interpolation
(defn ^:no-doc delete-tag [opts]
  (let [{:keys [::lock-name]} opts]
    (generic-cmd :opts opts
                 :cmd (format "git tag -d %s"
                              lock-name))))

After — one rebindable seam, argv vectors

;; run.clj
(defn default-runner [shell-opts cmd]
  (process/shell shell-opts cmd))
(def ^:dynamic *runner* default-runner)

(defn generic-cmd [& {:keys [opts cmd key shell-opts]}]
  (let [shell-opts (merge {:continue true :out :string
                           :err :string} shell-opts)
        proc (*runner* shell-opts cmd) ; single seam
        opts (handle-cmd opts proc)]
    (if key
      (assoc opts key (-> (:out proc) str/trim-newline))
      opts)))

;; lock.clj — argv, no shell parsing
(defn ^:no-doc delete-tag [opts]
  (generic-cmd :opts opts
               :cmd ["git" "tag" "-d"
                     (::lock-name opts)]))

Now unit-testable without git

(deftest delete-tag-builds-correct-argv
  (binding [run/*runner* (fn [_ cmd] {:exit 0 :out "" :err "" :cmd cmd})]
    (is (= ["git" "tag" "-d" "LOCK-DEAD"]
           (-> (lock/delete-tag {::lock/lock-name "LOCK-DEAD"})
               ::bc/procs first :cmd)))))
Impact — Testability / Safety / DX: Makes ~half the codebase deterministically unit-testable (fast CI, no git/bash dependency), closes the injection/tokenization fragility class repo-wide, and yields a single auditable execution chokepoint for logging, dry-run, and sandboxing.
#3High

Hidden mutable atoms drive the “functional” engine; pluggable dispatch reflects on runtime internals

Location
workflow.clj:340–390 (run-steps); pluggable.clj:21–29 (->workflow*)
Dimensions
Architecture & Design · Maintainability · Testability

The issue. The engine already threads opts functionally and exposes next-fn for branching, yet run-steps layers two closed-over atoms (opts*, steps*) and drives iteration by swap!-ing a hidden queue inside next-fn. The remaining step list is invisible to opts, so step-fn middleware can't observe it, runs aren't inspectable or resumable, and the "pure step" contract the framework advertises is quietly broken. Separately, pluggable/->workflow* resolves the multimethod by reflecting on private clojure.lang.MultiFn internals (.getMethod) — twice per step — coupling the framework to runtime implementation details.

Before — workflow.clj

(defn run-steps [step-fns {:keys [::steps] :as opts}]
  (let [opts*  (atom opts)        ; hidden state
        steps* (atom (map ... steps))
        wf (pluggable/->workflow*
            {... :next-fn
             (fn [step _ {:keys [::bc/exit] :as o}]
               (reset! opts* o)                ; mutate
               (cond
                 ...
                 :else
                 (let [n (first @steps*)
                       _ (swap! steps* rest)]   ; mutate
                   ...)))})]
    (wf step-fns @opts*)))

After — queue lives in opts; engine stays pure

(defn run-steps [step-fns {:keys [::steps] :as opts}]
  (let [opts (assoc opts ::queue
                    (map #(if (namespace %) %
                              (keyword "big-config.workflow"
                                       (name %)))
                         steps))
        wf (pluggable/->workflow*
            {:first-step ::start :last-step ::end
             :wire-fn (fn [step sfs] ...)
             :next-fn
             (fn [step _ {:keys [::bc/exit ::queue]
                          :as o}]
               (cond
                 (= step ::end) [nil o]
                 (pos? exit)    [::end o]
                 :else
                 (if-let [n (first queue)]
                   [n (assoc o ::queue (rest queue))]
                   [::end o])))})]
    (wf step-fns opts)))

Before — pluggable.clj (reflection)

(let [method
      (.getMethod ^clojure.lang.MultiFn
                  handle-step step)
      default-method
      (.getMethod ^clojure.lang.MultiFn
                  handle-step :default)]
  (if (and method (not= method default-method))
    (handle-step f step step-fns opts)
    (f opts)))

After — dispatch via the multimethod itself

(defmulti handle-step
  (fn [_f step _sfs _opts] step))

(defmethod handle-step :default [f _ _ opts]
  (f opts))

;; wire-fn just wraps the step:
(fn [step sfs]
  (let [[f next-step] (wire-fn step sfs)]
    [#(handle-step f step sfs %) next-step]))
Impact — Maintainability / Robustness: The workflow becomes fully data-driven (inspectable, resumable, observable by step-fns) and thread-safe by construction; the framework no longer depends on clojure.lang.MultiFn internals, removing a silent breakage point across Clojure upgrades and roughly halving the cognitive load of the engine's core.
#4Medium

Duplication & dead code from an unfinished step → workflow deprecation

Location
selmer_filters.clj:6–12; print-step-fn in step.clj:22–57 & workflow.clj:284–318; step.clj, build.clj
Dimensions
Code Quality · Maintainability

The issue. Three concrete instances of avoidable drift: (1) the :lookup-env Selmer filter is registered twice, identically; (2) the ~35-line print-step-fn is duplicated near-verbatim between the deprecated big-config.step and the live big-config.workflow — two copies that must be kept in sync by hand; (3) big-config.step and big-config.build exist only to serve the deprecated DSL yet are still fully wired into the dependency graph. step.clj/parse additionally has a cond with no :else, so it can silently return nil.

Before — selmer_filters.clj

(add-filter! :lookup-env
             (fn [x] (System/getenv x)))

(add-filter! :lookup-env          ; dead
             (fn [x] (System/getenv x))) ; duplicate

After

(add-filter! :lookup-env
             (fn [x] (System/getenv x)))
  • Extract the single source-of-truth print-step-fn into big-config.step-fns and have both call sites reuse it (delete the step.clj copy).
  • Move big-config.step + big-config.build behind an explicit deprecated/ path, or delete once the last caller migrates — CLAUDE.md already forbids using build as a synonym for render.
  • Add an :else branch to step.clj/parse (or delete the namespace).
Impact — DX / Maintainability: Removes ~80 lines of duplicated/dead code and one entire class of "fixed it in one copy, not the other" bugs; shrinks the public namespace surface and clarifies which DSL is current.

Secondary Findings

Lower-priority items worth scheduling, not blocking.

AreaObservationRecommendation
Med Robustness render.clj:220–226 — Files/createTempDirectory + .deleteOnExit. deleteOnExit only removes an empty dir at JVM shutdown, so populated temp trees accumulate on disk in long-lived processes (REPL/CI). Wrap in try/finally with recursive fs/delete-tree.
Med Security lock.clj:92–98 — parse-tag-content runs edn/read-string on the body of a git tag fetched from a remote, then uses it for the ownership check. A trust boundary on attacker-influenceable data. Validate shape with a spec before trusting it; reject unexpected forms.
Med Complexity render.clj:200–241 (copy-template-dir) and workflow.clj:501–559 (->workflow*) carry very high cognitive complexity — nested case/if/run! and chained threading. Decompose into named sub-functions (one responsibility each); add unit tests around the seams.
Low Consistency Error returns are constructed ad hoc ((merge opts {::bc/exit 1 ::bc/err "…"})) in many steps, while success has a shared core/ok helper. Add a symmetric core/err helper and use it everywhere for uniform shape.
Low Maintainability workflow.clj opens with a ~260-line namespace docstring before any code; it also contains contradictory/aspirational examples. Move long-form docs to a doc/ guide (quickdoc can still ingest it); keep concise docstrings on the public vars.
Low Robustness core.clj:172–185 — ->step-fn validation only catches the exact [nil :same] pair and emits a confusing message. Validate each argument independently with a clear error per case.

Testability Deep-Dive

Coverage map. Pure, deterministic logic is well tested — workflow_test.clj exercises parse-args, select-globals, new-prefix, prepare, merge-params, read-bc-pars; pluggable, render, store, run have focused tests. The gap is everything that touches the outside world:

  • 11 of 21 namespaces have no unit test, including lock (git-tag locking — security-sensitive), git, unlock, system, step_fns, selmer_filters, toml, big-tofu/create.
  • The single integration test (integration_test.clj) creates real bare git repos and spawns a child clojure -M:test — accurate but slow and environment-coupled, and it is the only thing exercising the lock/git logic at all.
  • Root cause is structural and identical to Finding 2: there is no seam between the code and process/shell. The single highest-leverage testability investment is the *runner* indirection — it converts the entire lock/git/unlock surface from "integration-only" to "unit-testable" with no behavioral change.

What Is Working Well

  • Composable core. ->workflow / ->step-fn and the opts-threading model are a clean, idiomatic state machine; middleware (step-fns) is a genuinely good extension mechanism.
  • Correct concurrency in the store. store.clj implements WATCH/MULTI/EXEC optimistic locking with a bounded restore-retry and snapshotting — ACID reasoning is explicit and sound.
  • Disciplined naming conventions. -> constructors, ! side-effects, ? predicates, namespaced keywords — applied consistently.
  • Boundary validation. render uses clojure.spec to conform templates/transforms and fails fast with explain-data; exceptions are funneled into opts with structured ex-data.
  • Reflection hygiene. render.clj sets *warn-on-reflection*. Extending it to the other interop-touching namespaces is a cheap, worthwhile habit.
