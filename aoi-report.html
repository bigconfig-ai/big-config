Executive Summary

BigConfig is a small (≈3,000 LOC), well-conceived Clojure/Babashka workflow engine. Its core abstraction — pure opts → opts steps threaded by a data-driven ->workflow combinator with a multimethod extension point — is genuinely elegant and the right design for the problem.

A prior review cycle has already closed the previous top findings: the process boundary now has a rebindable *runner* seam with argv vectors, the pluggable reflection was removed, and the new-prefix / composition-layer docstrings were corrected. Those are explicitly acknowledged as resolved/by-design and not re-flagged here.

This report widens the lens to all remaining improvements. The headline new issue is a silent correctness bug: binary template files are corrupted because the renderer round-trips every file through slurp/spit as a String. The rest are accumulated structural debt — a fully duplicated deprecated namespace, a copy-pasted "function-or-symbol" coercion with diverging contracts, namespace-load side effects, and an inconsistent error model. None require a rewrite; each is a localized, low-risk fix.

Critical

1 · Binary template files are corrupted by a slurp/spit round-trip

PerformanceSecurity & RobustnessCorrectness
src/clj/big_config/render.clj  ·  copy-dir  L183–198

Issue. copy-dir computes a replacable flag that correctly excludes binary extensions (jpg jpeg png gif bmp bin) from Selmer rendering — but it still reads every file with (slurp path) and writes it with (spit target-file content). slurp decodes bytes to a Java String using the platform default charset; spit re-encodes that String back to bytes. For non-UTF-8 byte sequences this round-trip is lossy: any binary asset shipped inside a template (icons, .bin seeds, compiled artifacts) is silently corrupted in the generated project. The replacable guard was clearly meant to protect these files; it only suppresses Selmer, not the destructive String conversion. This is a correctness bug first, and a memory-footprint problem second (whole files are buffered as Strings rather than streamed).

Before

(defn ^:no-doc copy-dir
  [& {:keys [src-dir target-dir data delimiters]}]
  (fs/walk-file-tree src-dir
    {:visit-file
     (fn [path _]
       (let [path (str path)
             replacable ((complement *non-replaced-exts*)
                         (fs/extension path))
             target-file ...
             _ (fs/create-dirs (fs/parent target-file))
             ;; binary files ALSO slurped to a String:
             content (cond-> (slurp path)
                       (and data replacable)
                       (selmer data delimiters))]
         (spit target-file content)
         (->> (fs/posix-file-permissions path)
              (fs/set-posix-file-permissions target-file)))
       :continue)}))

After

(defn ^:no-doc copy-dir
  [& {:keys [src-dir target-dir data delimiters]}]
  (fs/walk-file-tree src-dir
    {:visit-file
     (fn [path _]
       (let [path (str path)
             replacable ((complement *non-replaced-exts*)
                         (fs/extension path))
             target-file ...
             _ (fs/create-dirs (fs/parent target-file))]
         (if (and data replacable)
           ;; text path: render, then write
           (spit target-file
                 (selmer (slurp path) data delimiters))
           ;; binary / no-data path: byte-exact copy
           (fs/copy path target-file
                    {:replace-existing true}))
         (->> (fs/posix-file-permissions path)
              (fs/set-posix-file-permissions target-file)))
       :continue)}))
Impact. Eliminates a silent data-corruption bug in every template that ships a binary asset (safety/correctness); fs/copy streams bytes instead of buffering whole files as Strings (memory); behavior for text templates is unchanged. Add a regression test that renders a fixture containing a known PNG and asserts a byte-for-byte (hash) match.

ArchitectureMaintainability
src/clj/big_config/step.clj  (whole ns)  ·  carryover of prior Finding 4 (still open)

(ns big-config.step

(defn run-steps

3 · "Function-or-symbol" coercion is duplicated 6× with diverging, partly unsafe contracts

Architecture (DRY)Robustness
core.clj L41–47 · workflow.clj L331–337 · render.clj L295–302, L335–338, L207–209 · git.clj L8–13
Status — Resolved on main. A single total helper big-config.utils/->fn (function → as-is; symbol/string → requiring-resolve; other ifn? → as-is; nil → caller default or a structured throw; else ex-info with :big-config/err-kind ::not-a-fn) now backs core/resolve-step-fns, workflow/resolve-fn and render's data-fn/template-fn/post-process-fn. The silent nil → NPE hole is closed (a string now resolves; non-coercible values throw a debuggable error). Regression test ->fn-test added; full suite green (32 tests / 82 assertions / 0 failures). Out of scope by design (different mechanisms, not value→fn coercion): copy-template-dir's s/conform tag dispatch (render.clj L207–209) and git/get-revision's string-vs-keyword opts indirection (git.clj L8–13).

Issue. The recurring need to accept "a fn, or a symbol to requiring-resolve, or nil" is open-coded at least six times, and the variants disagree on their contract:

  • core/resolve-step-fns accepts fn or string, else throws.
  • workflow/resolve-fn accepts fn or symbol, nil → throws.
  • render's data-fn/template-fn/post-process-fn cond handles nil/fn/symbol but has no :else → a string (the form core accepts!) silently yields nil, and the next call (data-fn opts) throws an opaque NullPointerException with no context.

So the same conceptual input is "string only" in one place, "symbol only" in another, and "silently nil" in a third. This is both a DRY violation and a latent robustness bug.

Before — render.clj (no :else)

data-fn (cond
          (nil? data-fn)    (fn [data _] data)
          (fn? data-fn)     data-fn
          (symbol? data-fn) (requiring-resolve data-fn))
;; string -> nil -> (data-fn opts) => NPE, no ctx

template-fn (cond
              (nil? template-fn)    (constantly edn)
              (fn? template-fn)     template-fn
              (symbol? template-fn) (requiring-resolve
                                      template-fn))
;; same silent-nil hole; repeated for post-process-fn,
;; workflow/resolve-fn, core/resolve-step-fns ...

After — one helper in big-config.utils

(defn ->fn
  "Coerce v to a function. Accepts an ifn, a symbol or
   string to requiring-resolve, or nil -> `default`.
   Throws ex-info (structured) on anything else."
  ([v] (->fn v nil))
  ([v default]
   (cond
     (nil? v)     default
     (ifn? v)     v
     (symbol? v)  (requiring-resolve v)
     (string? v)  (requiring-resolve (symbol v))
     :else (throw (ex-info "Cannot coerce to fn"
                           {:value v :type (type v)})))))

;; render.clj
data-fn     (->fn data-fn (fn [data _] data))
template-fn (->fn template-fn (constantly edn))
;; workflow/resolve-fn, core/resolve-step-fns -> ->fn
Impact. One tested, total function replaces six partial ones (DRY, maintainability); closes the silent-nil → NPE hole and gives every call site a structured, debuggable error consistent with the project's ex-info contract (robustness, DX); unifies the accepted input shape so docs can finally state one rule.

4 · Namespace-load side effects: a literal duplicate filter, a frozen env, and global registry mutation

Code QualityTestability
selmer_filters.clj L6–16 · big_tofu/core.clj L29 · system.clj L24–34

Issue. Three top-level side effects fire at require time:

  • Literal copy-paste bug: selmer_filters.clj registers :lookup-env twice with an identical body (L6–8 then L10–12). It is harmless only by luck (idempotent) and is a clear unreviewed paste.
  • (def env (read-system-env)) in system.clj snapshots the entire process environment once, at load. Tests cannot vary it; any code reading system/env is untestable without a JVM relaunch.
  • add-filter! calls in two namespaces mutate Selmer's global registry as an import side effect — order-dependent and impossible to scope per render.

Before

;; selmer_filters.clj — registered twice, verbatim
(add-filter! :lookup-env
             (fn [x] (System/getenv x)))
(add-filter! :lookup-env            ; <-- duplicate
             (fn [x] (System/getenv x)))

;; system.clj — frozen at require time
(def env (read-system-env))         ; tests can't vary

After

;; selmer_filters.clj — register once, in an
;; explicit, idempotent init fn
(defn register-filters! []
  (add-filter! :lookup-env (fn [x] (System/getenv x)))
  (add-filter! :->file (fn [n] ...)))
;; called once from a known entry point, not on require

;; system.clj — read on demand
(defn env
  ([] (read-system-env))
  ([k] (get (read-system-env) k))
  ([k d] (get (read-system-env) k d)))
Impact. Deletes a dead duplicate registration (code quality); makes environment-dependent code unit-testable by passing/overriding env instead of relaunching the JVM (testability); converts implicit, order-sensitive load-time mutation into an explicit, idempotent init step (predictability). Note: env is currently a def, so making it a fn is a breaking call-site change — schedule with a deprecation alias.

5 · Inconsistent error model: IllegalArgumentException bypasses the structured ex-data contract

ArchitectureRobustness
render.clj L289, L319–320, L328 · workflow.clj L556 · core.clj L131, L174–175

Issue. CLAUDE.md states the error contract: "Throw ex-info with structured ex-data… the workflow engine catches exceptions and merges ex-data into opts" — and core/try-f indeed does exactly that. But several boundaries throw bare IllegalArgumentException / ex-info with an empty map: render/render throws IllegalArgumentException for nil templates and :transform errors but ex-info (with data) for "resource not found"; ->workflow* throws IllegalArgumentException for a bad pipeline; ex-info "…" {} appears with no context. When an IllegalArgumentException reaches try-f it still produces ::bc/err, but no structured ex-data is merged, so downstream branching and error reports lose the context the contract promises.

Before — mixed

;; render.clj
(when (nil? templates)
  (throw (IllegalArgumentException.
          ":big-config.render/templates ... nil")))
...
(throw (ex-info "Template resource not found"
                {:template template}))   ; structured
...
(throw (ex-info (str target-dir " already exists ...")
                {}))                     ; empty ctx

;; workflow.clj
(throw (IllegalArgumentException.
        ":pipeline must be like [::tool/tofu ..."))

After — uniform ex-info

(when (nil? templates)
  (throw (ex-info
          ":big-config.render/templates is nil"
          {:big-config/err-kind ::missing-templates
           :opts opts})))
...
(throw (ex-info "Template resource not found"
                {:big-config/err-kind ::no-resource
                 :template template}))
...
(throw (ex-info (str target-dir " already exists")
                {:big-config/err-kind ::target-exists
                 :target-dir target-dir}))
Impact. Every failure now arrives at try-f carrying a machine-readable :big-config/err-kind + context, so next-fn branching and error step-fns can react precisely instead of string-matching messages (robustness); the codebase finally matches its own documented contract (architecture consistency, DX).

6 · Env-var → keyword mapping collides . and _ (last write wins)

Security & Robustness
workflow.clj read-bc-pars L685–693 · system.clj keywordize L13–17

Issue. Both env-ingestion paths normalize a variable name by replacing both _ and . with -. So BC_PAR_DB_HOST and a (less common but legal) BC_PAR_DB.HOST both map to :db-host; in system/read-system-env, FOO_BAR and FOO.BAR both become :foo-bar. Because the results are poured (into {}), a collision is resolved silently and by undefined map-iteration order — a config value can change meaning depending on unrelated environment entries. For a tool whose entire premise is reproducible config generation, a silent, order-dependent override is a real robustness hole.

Before

;; workflow/read-bc-pars
(-> k (subs (count prefix))
      str/lower-case
      (str/replace "_" "-")
      (str/replace "." "-")   ; _ and . both -> -
      keyword)
;; ... (into {})  -> collisions resolved arbitrarily

;; system/keywordize — identical hazard
(-> (str/lower-case s)
    (str/replace "_" "-")
    (str/replace "." "-")
    keyword)

After — detect collisions

(defn- env->kw [s]
  (-> (str/lower-case s)
      (str/replace "_" "-")
      (str/replace "." "-")
      keyword))

(defn- index-env [pairs]
  (reduce (fn [m [k v]]
            (let [kw (env->kw k)]
              (when (contains? m kw)
                (throw (ex-info "Ambiguous env var name"
                         {:keyword kw :colliding k})))
              (assoc m kw v)))
          {} pairs))
;; fail loud instead of silently dropping a value
Impact. Converts a silent, order-dependent config override into a loud, actionable error at ingestion (robustness, reproducibility); negligible cost; no behavior change for the common collision-free case.

7 · mktemp-create-dir shells bash -c 'readlink -f $(mktemp -d)' — non-portable & re-tokenized

RobustnessPortability
src/clj/big_config/run.clj  ·  mktemp-create-dir L45–50

Issue. This helper builds a temp dir by passing the string "bash -c 'readlink -f $(mktemp -d)'" to generic-cmd, which word-tokenizes it via babashka.process. Two problems: (1) it depends on bash + GNU readlink -f, which does not exist on stock macOS (BSD readlink has no -f) — and this project's own environment is darwin, so this path is broken on the maintainer's machine without coreutils; (2) it reintroduces the very string-tokenization the prior cycle eliminated elsewhere with argv vectors. The JVM already offers a portable, dependency-free primitive.

Before

(defn mktemp-create-dir [opts]
  (let [{:keys [::dir] :as opts}
        (generic-cmd
          :opts opts
          :cmd "bash -c 'readlink -f $(mktemp -d)'"
          :key ::dir)
        shell-opts {::shell-opts {:dir dir}}]
    (merge opts shell-opts)))

After — pure JVM, portable

(defn mktemp-create-dir [opts]
  (let [dir (-> (Files/createTempDirectory
                  "big-config"
                  (into-array FileAttribute []))
                .toRealPath          ; canonical, no readlink
                str)]
    (-> opts
        core/ok
        (assoc ::dir dir)
        (assoc-in [::shell-opts :dir] dir))))
Impact. Removes a hard dependency on bash/GNU readlink and fixes the helper on macOS (portability, robustness); drops one process spawn (perf); stays consistent with the project's "argv / no string tokenization" hardening direction.

8 · The Redis store boundary has no test seam (the *runner* pattern was not applied here)

Testability
src/clj/big_config/store.clj  ·  write! / restore! / store!

Issue. The prior cycle correctly introduced a rebindable *runner* var so the process boundary can be exercised without spawning real processes. The Redis boundary has the identical shape — write! and restore! are defn- and call carmine/wcar directly — but no equivalent seam, so every store test needs a live Redis. The event-sourcing logic (optimistic-lock retry, snapshot cadence, replay hash verification) is the most subtle code in the repo and is exactly what you most want to unit-test deterministically.

Before — hard-wired to carmine

(defn- write! [offset ts event hash key wcar-opts]
  (-> (wcar wcar-opts (car/watch key) ...) ...))

(defn- restore! [handler state-atom key wcar-opts]
  (let [states (wcar wcar-opts (car/zrange ...))]
    ...))
;; store-test.clj requires a running redis-server

After — a journal protocol seam

(defprotocol Journal
  (j-append! [_ offset ts event hash])
  (j-range   [_ lo hi]))

(defn redis-journal [wcar-opts key] ...)   ; prod impl
(defn atom-journal  []                     ; test impl
  (let [a (atom (sorted-map))] ...))

;; store! takes a Journal; write!/restore! call the
;; protocol, not carmine. Unit tests use atom-journal;
;; one integration test still hits real Redis.
Impact. Makes the CAS/retry/snapshot/replay logic unit-testable without infrastructure (testability, CI speed/reliability); applies the seam pattern the team already adopted for processes, keeping the codebase consistent; an in-memory atom-journal also doubles as a zero-dependency local mode.

9 · :overwrite true silently merges over an existing target directory

Robustness
src/clj/big_config/render.clj  ·  render L324–328

Issue. When the target dir exists: :overwrite falsey → throw; :overwrite :delete → wipe then write. But :overwrite true takes neither branch — it falls through and copies on top of whatever is already there. Stale files from a previous render that are no longer produced by the template silently survive into the "regenerated" output, undermining the reproducibility guarantee. (The behavior mirrors deps-new, but here the docstring does not warn callers.)

Before

(when (.exists (io/file target-dir))
  (if overwrite
    (when (= :delete overwrite)
      (b/delete {:path target-dir}))
    (throw (ex-info (str target-dir
              " already exists ...") {}))))
;; overwrite=true: no delete, no throw -> merge-over

After

(when (.exists (io/file target-dir))
  (case overwrite
    :delete (b/delete {:path target-dir})
    true    nil   ; documented: intentional merge-over
    (throw (ex-info (str target-dir " already exists; "
                "pass :overwrite :delete to replace")
              {:big-config/err-kind ::target-exists
               :target-dir target-dir}))))
;; + document the merge-over semantics in the docstring
Impact. Makes the three overwrite modes explicit and documented so callers can choose :delete for true reproducibility and aren't surprised by surviving stale files (robustness, DX). Low severity: behavior may be intended — the fix is primarily making the contract explicit.

11 · port-assigner uses hash-mod with no collision detection

ScalabilityRobustness
src/clj/big_config/utils.clj  ·  port-assigner L29–35

Issue. Ports are derived as (+ 1024 (mod (abs (hash [cwd service])) 64000)). This is deterministic (good for the local-dev use case) but has no collision check: by the birthday bound, a few dozen services in one project have a non-trivial chance of two getting the same port, surfacing only as a confusing bind/connect failure at process-compose time.

Before

(defn port-assigner [service]
  (-> (fs/cwd) (str service) hash abs
      (mod 64000) (+ 1024)))
;; two services -> possibly same port, no signal

After — keep determinism, add probing

(defn port-assigner
  ([service] (port-assigner service #{}))
  ([service taken]
   (loop [p (-> (fs/cwd) (str service) hash abs
                (mod 64000) (+ 1024))]
     (if (contains? taken p)
       (recur (inc (if (< p 65535) p 1023)))
       p))))
;; caller threads the set of already-assigned ports
Impact. Preserves deterministic-by-default assignment while making multi-service projects collision-free and predictable (robustness). Low severity: only bites projects with many co-located services.
Low

12 · store/restore! performs a full event replay on every write-contention retry

Performance & Scalability
src/clj/big_config/store.clj  ·  handle! → restore! L107–127, L58–72

Issue. On a failed optimistic write (or a no-op business fn), handle! calls restore!, which issues car/zrange … "+inf" — reading every event from the current offset to the end of the journal and re-applying them. Under write contention this is O(events) work per retry, and snapshot pruning only caps history every snapshot-every writes. For a long-lived store under concurrent writers the retry cost grows with journal length rather than with the number of missed events.

Before

;; restore! always pulls to +inf and replays all
(car/zrange store-key start-index "+inf" "BYSCORE")
;; called on every CAS-fail / no-op retry in handle!

After — bounded incremental catch-up

;; fetch only up to the offset we actually need, in
;; capped batches, and stop once caught up:
(car/zrange store-key start-index
            (str target-offset) "BYSCORE"
            "LIMIT" "0" batch-size)
;; loop batches until current; avoids full-tail scan
;; on every contention retry
Impact. Makes retry cost proportional to missed events, not total journal length (scalability under contention). Low severity today (no evidence of high-concurrency use), but cheap insurance before the store is used at scale.

Acknowledged as By-Design / Already Resolved — Not Re-flagged

A prior review cycle (visible in the git history and codified in CLAUDE.md / the namespace docstrings) addressed the previous top findings. Per the project's explicit guidance, these are not re-raised — they are recorded here only so this report is a complete picture and to confirm they were respected:

  • By design Closed-over atoms in run-steps / ->workflow*. These are the deliberate accumulator + subworkflow-isolation barrier of the composition layer, not a purity leak. CLAUDE.md "What to Avoid" forbids "de-atomizing" them; the docstrings explain the invariant. Left unchanged by design.
  • Resolved Process boundary / command injection. The rebindable *runner* seam and argv-vector internal commands are in place (run.clj, lock.clj, git.clj) with lock_test.clj exercising the lock workflow without git. (Finding 7 above is a different, residual string-tokenized call the hardening missed, not a regression of this.)
  • Resolved pluggable MultiFn reflection. Removed in favor of an explicit :default method + direct dispatch (commit 0c5b11e).
  • By design new-prefix determinism. Deterministic paths are an intentional requirement (local-backend state continuity, create/delete pairing); concurrency is an explicit non-goal delegated to the lock step. Docstring already narrowed.
