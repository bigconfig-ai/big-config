BigConfig

A Clojure/Babashka workflow and template engine for infrastructure-as-code automation — a zero-cost build step that uses Clojure to generate config files and orchestrate CLI tools (Terraform/OpenTofu, Ansible, Kubectl, …).

Overview #

BigConfig bridges general-purpose programming (Clojure/Babashka) and specialized CLI tools. It composes small, independently testable tool workflows into pipelines, renders configuration via a Selmer-based template engine, and coordinates execution with a client-side Git-tag locking mechanism. The library is consumed as a Clojure tool (clojure -Ttools) or via Babashka tasks (bb).

Workflow

State-driven engine that threads an opts map through pluggable steps.

Render

Selmer-based template engine inspired by deps-new.

Lock

Client-side Atlantis: pessimistic locking via Git tags.

Store

Redis-backed event-sourcing store with ACID-like guarantees.

System

Lifecycle management alternative to Integrant.

BigTofu

Clojure constructs for generating OpenTofu/Terraform HCL.

Why BigConfig #

  • Zero-cost build step: use Clojure to generate JSON/YAML/HCL config, then execute tools seamlessly.
  • Functional orchestration: sequence multiple tools into a unified lifecycle (render → lock → tofu apply → unlock).
  • Serverless locking: client-side, Git-based locks coordinate team work without a central server like Atlantis.
  • Extreme portability: develop and debug locally with a REPL, then run the exact same workflow in CI with parity.
  • Decoupled data sharing: orchestrators map outputs of one tool into params of the next, so a Hetzner workflow can be swapped for AWS without changes to downstream Ansible.

Installation #

BigConfig is typically used as a Clojure tool or via Babashka.

# Install BigConfig as a Clojure tool
clojure -Ttools install-latest :lib io.github.amiorin/big-config :as big-config

# Print help for all available templates
clojure -A:deps -Tbig-config help/doc

# Scaffold a new project using the `package` template
clojure -Tbig-config package \
  :owner acme :repository infra :ssh-key 123456 :target-dir my-infra

The published artifact name is motain/big-login (see build.clj). Versions follow 0.3.<git-commit-count>.

The opts map #

Every workflow function receives and returns an opts map. The engine threads it through each step. Reserved top-level keys live in the big-config namespace (commonly aliased as bc):

KeyTypeMeaning
:big-config/exitnat-int?Exit code (0 = success). Required after every step.
:big-config/errstring?Error message.
:big-config/stack-tracestring?Exception stack-trace (joined newline-separated).
:big-config/env:repl, :lib, :shellSelects shell-output behavior and exit handling.
:big-config/procsvectorAccumulated child-process results.
:big-config/stepsvectorSteps recorded by log-step-fn.

Minimal step

(ns my.app
  (:require [big-config.core :as core]))

(defn my-step [opts]
  ;; do work…
  (core/ok opts))            ; sets :big-config/exit 0

(defn failing-step [opts]
  (merge opts {:big-config/exit 1
               :big-config/err  "Something went wrong"}))

The engine catches exceptions thrown by step functions, merges ex-data into opts, and sets :big-config/exit 1 with the exception message and stack trace. Throw ex-info for structured errors that should propagate.

An opts map is what the engine threads forward. If a step returns nil or sets a non-natural :big-config/exit, the engine throws — contracts are enforced.

CLI DSL #

BigConfig provides a concise DSL for running workflows from a shell via Babashka:

bb render lock tofu:init tofu:apply -- tofu apply -auto-approve
#  │      │    │                       └── raw shell command
#  step   step colon-syntax (tofu init)
  • Steps (e.g. render, lock) are predefined workflow names.
  • Colon syntax: tool:subcommand is rewritten to tool subcommand for shell execution.
  • Separator --: everything after it is forwarded as one raw shell command to exec.

Examples

# exec vs. direct mapping (functionally identical)
bb exec -- ansible-playbook main.yml
bb ansible-playbook:main.yml

# Raw command after `--`
bb render -- tofu plan -out plan.tfplan

# Inline arguments (per command)
bb tofu:apply:-auto-approve tofu:destroy:-auto-approve

# Tool mapping
# tofu:init                  -> tofu init
# tofu:plan                  -> tofu plan
# ansible-playbook:main.yml  -> ansible-playbook main.yml

Zero-cost workflow alias

You can hide BigConfig behind a shell alias that wraps your standard tools in a safety-first pipeline:

alias tofu="bb render git-check lock exec git-push unlock-any -- tofu"

Argument parsing

The DSL is parsed by workflow/parse-args:

(workflow/parse-args "render")
;; => {:big-config.workflow/steps    [:render]
;;     :big-config.run/cmds          []}

(workflow/parse-args ["render" "lock" "tofu:init" "tofu:apply" "--" "tofu" "apply" "-auto-approve"])
;; => {:big-config.workflow/steps    [:render :lock :exec]
;;     :big-config.run/cmds          ["tofu init" "tofu apply" "tofu apply -auto-approve"]}

The set of recognized step keywords lives in a dynamic var workflow/*parse-args-steps*. Default value: #{:lock :git-check :render :create :delete :exec :git-push :unlock-any}. Unrecognized tokens are treated as shell commands and routed through the implicit :exec step.

Workflow engine #

At the heart of BigConfig is big-config.core/->workflow. It produces a function that, given step functions (middleware) and an opts map, loops through steps, threads opts, and decides the next step.

->workflow

(ns my.app
  (:require [big-config.core :as core]))

(core/->workflow
  {:first-step ::start          ; required — qualified keyword
   :last-step  ::end            ; optional, defaults to `::end` in the same ns
   :wire-fn    (fn [step step-fns]
                 (case step
                   ::start [my-fn ::end]
                   ::end   [identity]))
   :next-fn    nil})            ; optional — for branching

The wire-fn receives the current step and the resolved step-fns, returning [step-impl next-step]. The default next-fn chooses the next step on success and jumps to last-step on failure, using core/choice.

Branching with next-fn

Use choice to branch on exit:

(def lock
  (core/->workflow
    {:first-step ::generate-lock-id
     :wire-fn (fn [step _]
                (case step
                  ::generate-lock-id [generate-lock-id ::delete-tag]
                  ::delete-tag       [delete-tag ::create-tag]
                  ::create-tag       [create-tag ::push-tag]
                  ::push-tag         [push-tag   ::get-remote-tag]
                  ::get-remote-tag   [(comp get-remote-tag delete-tag) ::read-tag]
                  ::read-tag         [read-tag   ::check-tag]
                  ::check-tag        [check-tag  ::end]
                  ::end              [identity]))
     :next-fn (fn [step next-step opts]
                (case step
                  ::end       [nil opts]
                  ::push-tag  (core/choice {:on-success ::end
                                            :on-failure next-step
                                            :opts opts})
                  ::delete-tag [next-step opts]
                  (core/choice {:on-success next-step
                                :on-failure ::end
                                :opts opts})))}))

Invocation forms

A workflow function supports two invocation forms:

(wf)                    ; => [first-step last-step]
(wf step-fns opts)      ; run the workflow

Step functions (middleware) #

A step function is middleware around an individual step. It receives the step implementation function, the step keyword, and opts — and must return an updated opts.

core/->step-fn is a convenience constructor that accepts :before-f and :after-f side-effect callbacks of arity [step opts]. These are typically used for logging, terminating the process, or printing errors. 

(require '[big-config.core :as core])

(def log-fn
  (core/->step-fn
    {:before-f (fn [step opts] (println "→" step))
     :after-f  (fn [step opts] (println "✓" step))}))

Built-in step functions (big-config.step-fns)

FunctionPurpose
->exit-step-fnWhen the workflow reaches the supplied end step and :big-config/env is not :repl, calls System/exit with the workflow exit code.
->print-error-step-fnOn non-zero exit at end, prints the error in red and writes the stack-trace to /tmp/big-config-….txt.
log-step-fnAppends each step keyword to :big-config/steps for later inspection.
bling-step-fnColored step-progress reporter via paintparty/bling.
tap-step-fnEmits [step :before/:after opts] tuples through tap> — handy with the debug macro.

Note: step-fns are composed in reverse order via the engine, so the first step-fn in the vector ends up outermost.

Pluggable steps (multimethod) #

big-config.pluggable provides ->workflow*, a drop-in replacement that wraps each step in a multimethod lookup. Whenever the handle-step multimethod has a defmethod for a given step keyword, it replaces the implementation provided by wire-fn. Otherwise the original function runs. 

(require '[big-config.pluggable :as pluggable]
         '[big-config.workflow  :as workflow])

(defmethod pluggable/handle-step ::my-step
  [f step step-fns opts]
  (println "Custom step!" step (count step-fns))
  (f opts))

;; Register the step keyword so the DSL parser recognizes it
(binding [workflow/*parse-args-steps*
          (conj workflow/*parse-args-steps* :my-step)]
  (workflow/parse-args ["my-step" "render"]))
Tip Use remove-method to unhook a custom step, e.g. while testing: 
(remove-method pluggable/handle-step ::lock)

Tool workflow

Tool workflows wrap render, exec, git-*, and (un)lock steps for one tool. They are defined in two layers — the library function and the Babashka entry point:

(ns wf
  (:require [big-config.render   :as render]
            [big-config.workflow :as workflow]))

(defn tofu
  [step-fns opts]
  (let [opts (workflow/prepare
               {::workflow/name ::tofu
                ::render/templates [{:template "tofu"
                                     :overwrite true
                                     :transform [["tofu" :raw]]}]}
               opts)]
    (workflow/run-steps step-fns opts)))

(defn tofu*
  [args & [opts]]
  (let [opts (merge (workflow/parse-args args) opts)]
    (tofu [] opts)))

And wired up in bb.edn:

{:deps {group/artifact {:local/root "."}}
 :tasks
 {:requires ([wf :as wf])
  tofu     {:doc  "bb tofu render tofu:init -- tofu apply -auto-approve"
            :task (wf/tofu* *command-line-args*)}
  ansible  {:doc  "bb ansible render -- ansible-playbook main.yml"
            :task (wf/ansible* *command-line-args*)}
  resource {:doc  "bb resource create"
            :task (wf/resource* *command-line-args*)}}}

Composite workflow

Composite workflows orchestrate multiple tool workflows through a :pipeline of [tool-keyword [args opts-fn]] pairs. The opts-fn is the glue: it inspects outputs of previous steps (for example by running tofu show --json) and maps them into ::workflow/params for the next tool — keeping downstream tools (Ansible) decoupled from upstream providers (Tofu, AWS, Hetzner, …). 

(defn opts-fn [opts]
  (let [ip (-> (p/shell {:dir (workflow/path opts ::tool/tofu)
                         :out :string}
                        "tofu show --json")
               :out
               (json/parse-string keyword)
               (->> (s/select-one
                     [:values :root_module :resources
                      s/FIRST :values :ipv4_address])))]
    (merge-with merge opts {::workflow/params {:ip ip}})))

(def resource-create
  (workflow/->workflow*
   {:first-step ::start-create-or-delete
    :last-step  ::end-create-or-delete
    :pipeline   [::tool/tofu          ["render tofu:init -- tofu apply -auto-approve"]
                 ::tool/ansible       ["render -- ansible-playbook main.yml" opts-fn]
                 ::tool/ansible-local ["render -- ansible-playbook main.yml" opts-fn]]}))

(def resource-delete
  (workflow/->workflow*
   {:first-step ::start-create-or-delete
    :last-step  ::end-create-or-delete
    :pipeline   [::tool/tofu ["render -- tofu destroy -auto-approve"]]}))

(defn resource
  [step-fns opts]
  (let [opts (merge {::workflow/create-fn resource-create
                     ::workflow/delete-fn resource-delete}
                    opts)
        wf (core/->workflow
            {:first-step ::start
             :wire-fn (fn [step step-fns]
                        (case step
                          ::start [(partial workflow/run-steps step-fns) ::end]
                          ::end   [identity]))})]
    (wf step-fns opts)))
Note ->workflow* calls workflow/new-prefix on the globals derived from opts, stamping a unique hash into both ::prefix and ::object-prefix. Each invocation writes to an isolated directory and object path, avoiding collisions when the same composite workflow runs concurrently.

run-steps dispatch

The dynamic workflow built by workflow/run-steps iterates the user-provided ::workflow/steps vector and dispatches each step to its implementation. Steps without a namespace are auto-qualified to big-config.workflow.

(workflow/run-steps
  step-fns
  {::workflow/steps      [:render :lock :exec :unlock-any]
   ::run/cmds            ["tofu init" "tofu apply" "tofu apply -auto-approve"]
   ::workflow/create-fn  my-create-fn
   ::workflow/delete-fn  my-delete-fn})

Render (template engine) #

big-config.render is a Selmer-based template engine inspired by seancorfield/deps-new. A minimal template consists of a resource path, a target directory, and one or more transforms: 

{:big-config.render/templates
 [{:template   "resource-path"
   :target-dir "target"
   :transform  [["."]]}]}

By default, render copies the source folder to :target-dir and replaces any {{key-a}} in filenames and contents with the corresponding :key-a from the data map.

Map glossary

NameMeaning
optsThe BigConfig map threaded through the workflow.
dataContext passed to Selmer (unless :raw).
ednThe high-level rendering config (one entry of ::templates).
transformA tuple [src target files delimiters opts] describing one transform.
filesMapping from template filenames to rendered target names.
delimitersOverride Selmer delimiters (avoids conflicts with target syntax).
transform-opts:only and :raw modifiers.

edn keys

KeyRequiredMeaning
:templaterequiredResource path with template files.
:target-dirrequiredWhere the rendered files are written (e.g. .dist).
:transformrequiredSequence of transforms.
:overwriteoptionaltrue or :delete — same as deps-new.
:data-fnoptional(fn [data opts] data) — modify the data map.
:template-fnoptional(fn [data edn] edn) — modify the edn map.
:post-process-fnoptionalFunction(s) run after copying, with (edn data).

Transform shapes

The :transform tuple follows [src target files delimiters opts].

;; src can be a folder or symbol/function
{:transform
 [['ansible/render-files
   {:inventory "inventory.json"
    :config    "default.config.yml"}
   :raw]]}

;; target, files, custom delimiters
{:transform
 [["src" "target"
   {"config.json" "config.json"}
   {:tag-open    \<
    :tag-close   \>
    :filter-open \<
    :filter-close \>}]]}

;; :only — copy just the listed files
{:transform
 [["src" "src/{{data-key-a}}"
   {"main.clj" "{{data-key-b|selmer-filter:param-a:param-b}}.clj"}]
  ["test" "test/{{data-key-b}}"
   {"main_test.clj" "{{data-key-c}}_test.clj"}
   :only]]}

;; :raw — disable Selmer rendering for a directory
{:transform
 [["resources" "resources/{{data-key-a}}"]
  ["templates" "resources/{{data-key-b}}/templates" :raw]]}

Built-in render step

The workflow version is exposed as render/templates, ready to drop into a workflow:

(def templates
  (core/->workflow {:first-step ::start
                    :wire-fn (fn [step _]
                               (case step
                                 ::start [render ::end]
                                 ::end   [identity]))}))

workflow/prepare additionally injects :target-dir and :target-object into every template map, so templates can reference them without repeating path logic.

Whitespace control workaround

Selmer does not natively support whitespace-trimming delimiters, so BigConfig implements {{- / -}} via a pre-processing pass (see selmer-filters/whitespace-control). Use it like this: 

{{- some-var -}}    # trims surrounding whitespace

Binary files

The dynamic var render/*non-replaced-exts* lists extensions that bypass Selmer: jpg jpeg png gif bmp bin. Rebind to extend.

Lock / Unlock #

BigConfig Lock provides Atlantis-like locking — but entirely client-side, using Git tags as the backing store. This restores interactive operation by removing mandatory pull-request workflows, and works equally well with Terraform, Ansible, and any other tool.

Required opts keys

KeyRequiredMeaning
:big-config.lock/owner required String distinguishing the user / CI environment.
:big-config.lock/lock-keys required Sequence of keys whose values uniquely identify the resource to lock.

Behavior

  • The lock id is the hex hash of the selected lock-keys values.
  • Re-acquiring a lock with the same owner succeeds (idempotent).
  • Acquiring a lock held by a different owner fails with exit 1 and a message "Different owner".

Steps

;; big-config.lock/lock pipeline
::generate-lock-id → ::delete-tag → ::create-tag → ::push-tag
       → on push success: done
       → on push failure: ::get-remote-tag → ::read-tag → ::check-tag

;; big-config.unlock/unlock-any pipeline
::generate-lock-id → ::delete-tag → ::delete-remote-tag → ::check-remote-tag

Usage

(require '[big-config.lock :as lock])

(lock/lock {::lock/owner     "alberto"
            ::lock/lock-keys [::env ::region]
            ::env            "prod"
            ::region         "eu-west-1"})

;; force release
(require '[big-config.unlock :as unlock])
(unlock/unlock-any {::lock/owner     "alberto"
                    ::lock/lock-keys [::env ::region]
                    ::env            "prod"
                    ::region         "eu-west-1"})

Run (shell execution) #

big-config.run wraps babashka.process/shell:

  • generic-cmd — runs a single command, captures stdout/stderr, merges results into opts.
  • run-cmd — runs one command from ::cmds using ::shell-opts; output handling depends on ::bc/env.
  • run-cmds — workflow that runs all of ::cmds sequentially, short-circuiting on failure.
  • handle-cmd — strips ANSI escape sequences, accumulates a :big-config/procs trace.
  • mktemp-create-dir / mktemp-remove-dir — create and clean throw-away dirs.

Environment behavior:

:big-config/envDefault :out / :err
:libcaptured as strings
:repl / :shellinherited from the parent process
 
(require '[big-config.run :as run])

(run/run-cmds
  [(fn [f step opts] (println step) (f step opts))]
  {::bc/env :repl
   ::run/shell-opts {:continue true
                     :dir "big-infra"
                     :extra-env {"FOO" "BAR"}}
   ::run/cmds ["bash -c 'echo Error >&2 && exit 1'"]})

Git helpers #

big-config.git/check is a workflow that asserts the local HEAD matches the upstream:

::git-diff → ::fetch-origin → ::upstream-name → ::pre-revision
          → ::current-revision → ::origin-revision → ::compare-revisions

If the working directory is dirty, the upstream is missing, or the local revisions diverge from origin, the workflow exits with code 1 and "The local revisions don't match the remote revision".

big-config.git/git-push is a thin wrapper around git push, surfaced as the git-push step.

Store (Redis event-sourcing) #

big-config.store is a Redis-backed event-sourcing store — a fork of prevayler-clj ideas. It journals each event in a Redis sorted set, replays them on restart, and provides ACID-like guarantees:

  • Atomicity — successful WATCH/MULTI/EXEC writes the event and bumps state atomically; failures retry.
  • Consistency — guaranteed by the business function; throw to abort.
  • Isolation — locking this serializes handle! calls.
  • Durability — events live in Redis with a per-event hash for replay verification.

API

FunctionPurpose
store!Constructor; returns a Store.
handle!Journals an event, applies the business fn, returns the new state.
snapshot!Persists a snapshot (negative offset) and trims older events.
restoreReplays missing events into the in-memory state.
get-offsetCurrent event offset.
deref (@store)Returns the current user state.
 
(require '[big-config.store :as store])

(def s
  (store/store!
    {:initial-state {:cnt 0}
     :business-fn   (fn [state {:keys [op]} _ts]
                      (case op :inc (update state :cnt inc)))
     :store-key     "store-state"
     :snapshot-every 10
     :wcar-opts     {:pool (car/connection-pool {})
                     :spec {:uri "redis://localhost:6379/"}}}))

(store/handle! s {:op :inc}) ; => {:cnt 1}
(store/snapshot! s)
(.close ^java.io.Closeable s)

The default constructor takes care of replay on startup. Snapshots are stored at -offset so a ZRANGEBYSCORE -inf 0 finds them ahead of regular events.

System (lifecycle) #

big-config.system is a workflow-based alternative to Integrant. Components are steps, and each step that starts a resource calls add-stop-fn to register cleanup. The terminal stop step iterates the accumulated ::stop-fns.

Helpers

FunctionPurpose
envEnvironment-variable map, keys keywordized (_ and . → -, lowercased).
destroy!Gracefully destroy a babashka.process/process with a timeout, then force-kill.
re-processSpawn a child process and block until :regex matches its :out/:err, or :timeout.
add-stop-fnRegister a cleanup function for later stop.
stopWorkflow step: invokes registered stop functions in order. Honors ::async to defer.
stop!Invokes the deferred stop function on an async system.

Example

(defn background-process [opts]
  (let [re-opts {:cmd     "clj -X my.app/main :shutdown-timeout 100"
                 :regex   #"token"
                 :timeout 3000
                 :key     ::proc}
        opts    (re-process re-opts opts)]
    (add-stop-fn opts
      (fn [{:keys [::proc] :as opts}]
        (when proc (destroy! proc 1000))))))

(def ->system
  (core/->workflow
    {:first-step ::start
     :wire-fn (fn [step _]
                (case step
                  ::start [background-process ::end]
                  ::end   [stop]))}))

;; synchronous (default): blocks until stop
(->system [log-step-fn] {::bc/env :repl})

;; asynchronous — stop later
(def system (->system [log-step-fn] {::bc/env :repl ::async true}))
(stop! system)

BigTofu (OpenTofu/Terraform constructs) #

big-tofu.core defines a To protocol and a Construct record for assembling HCL via Clojure data. A Construct can be referenced by property (Terraform-style interpolation) or converted into an ARN string. 

(require '[big-tofu.core :as t]
         '[big-tofu.create :as tc])

(def kms-blocks  (tc/kms :alpha/big-kms))
(def sqs-blocks  (tc/sqs :alpha/big-sqs))
(def buckets     (tc/bucket :alpha/big-bucket "foo" "bar"))

(->> kms-blocks
     (map t/construct)
     (apply utils/deep-merge)
     utils/sort-nested-map
     pp/pprint)

The stdlib (big-tofu.create) provides convenience builders:

  • bucket — variadic, accepts a base FQN and optional suffix tokens.
  • sqs — single SQS queue.
  • kms — KMS key with an IAM policy permitting the AWS account root.
  • provider — AWS + S3 backend stanza, optional assume_role.

BC_PAR_ overrides #

Override any project parameter from CI by prefixing the env var with BC_PAR_. The prefix is stripped, the rest is lower-cased, underscores and dots become dashes, and the result is converted to a keyword inside ::workflow/params. 

export BC_PAR_CLOUDFLARE_ZONE_ID="your-zone-id"
# becomes {:cloudflare-zone-id "your-zone-id"} in ::workflow/params

This is applied by workflow/read-bc-pars, which merges the env-derived params with the existing ones.

The debug macro

debug captures everything sent through tap> during the body and exposes it both as a sorted snapshot bound to a Var and as an extra key on the result (when the result is a map). 

(require '[big-config.utils :refer [debug]])

(debug tap-values
  (defn fn-wip []
    (tap> [:fn-wip {::b {::d 4 ::c 3} ::a 1}])
    {::b {::d 4 ::c 3} ::a 1})
  (fn-wip))
;; tap-values is now def'd in the current ns as a sorted vector
;; of every tap> emitted inside the body.

Even when the body throws, tap-values is still populated for inspection. LSP indentation for the macro is configured in .lsp/config.edn.

Selmer extras #

big-config.selmer-filters registers two filters and exposes whitespace-control:

  • :lookup-env — reads an environment variable.
  • :->file — replaces . with / and - with _, suitable for converting a namespace into a file path.
  • whitespace-control — pre-processor that implements {{- / -}} and {%- / -%} (and the matching closers) by trimming surrounding whitespace before Selmer runs.
;; Inside a template
;; {{ "PATH"|lookup-env }}            → reads $PATH
;; {{ "big-config.core"|->file }}     → "big_config/core"

CI/CD #

.github/workflows/ci.yml triggers on push to main (ignoring **/*.md and /.gitignore changes):

Conventions #

Naming

PatternMeaningExample
-> prefixConstructor->workflow, ->step-fn
! suffixSide-effecting / destructivewrite!, restore!, destroy!
? suffixPredicate(ifn? f)
:: qualified keywordsNamespaced options::bc/exit, ::lock/owner
^:privatePrivate var(def ^:private prefix "BC_PAR_")
[name] vs [name]*Library fn vs Babashka tasktofu takes step-fns + opts; tofu* takes args

Error handling

  • Throw ex-info with structured ex-data.
  • The workflow engine catches exceptions and merges ex-data + :big-config/err + :big-config/stack-trace into opts.
  • Validate at system boundaries; trust internal function contracts.

Specter

com.rpl/specter drives complex nested-data transforms (e.g., MAP-WALKER, deep-sort-maps in big-config.utils). Specter macros are mapped under .clj-kondo/config.edn for correct linting.

What to avoid #

Removed namespaces — do not use aero/aero, big-config.aero, big-config.call, big-config.clone, big-config.package were removed.
  • Don't pass step-fns inside the ->workflow map — pass it as the first argument when invoking the workflow.
  • Don't use build as a synonym for render — render is the current name (the old deps-new integration was renamed).
  • Don't modify .github/workflows/ci.yml to skip tests — the tag job depends on test success.
  • Don't push to main directly; everything goes through feature branches and CI.

Glossary #

opts map
The BigConfig map threaded through every workflow step.
step
A qualified keyword naming a node in a workflow's state graph.
step function
Middleware wrapping a step. Built via ->step-fn or written by hand [f step opts].
tool workflow
The fundamental unit: render + execute one CLI tool.
comp workflow
Sequences tool workflows for create / delete lifecycles.
system workflow
Lifecycle orchestrator for background components (see big-config.system).
profile
Convention for environment identifiers (dev, prod, …). Available to templates as :profile.
module
Convention for monorepo sub-projects (borrowed from Integrant). Available to templates as :module.
prefix / object-prefix
Base directory and object naming. Defaults: .dist and tofu. Stamped with a unique hash by workflow/new-prefix so concurrent invocations don't collide.
lock-id
Hex hash of the values selected by :big-config.lock/lock-keys — uniquely identifies a locked resource.
